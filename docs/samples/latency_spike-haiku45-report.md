# Incident investigation: latency_spike

**Severity: SEV-1** (confidence high) — Error rate on booking-service POST /book reached 4.8% at 2026-06-01T14:40:00+00:00 (baseline 0.3%), exceeding the 1-25% SEV-2 range and approaching critical threshold. P95 latency reached 3200ms at 2026-06-01T14:40:00+00:00 (baseline 450ms), a 7.1x increase. Multiple booking requests exhausted retry logic and failed entirely (events at 14:35:52, 14:38:09, 14:45:17), indicating users cannot complete the critical appointment booking flow. The incident remains unresolved as of 2026-06-01T15:00:00+00:00 with sustained error rate ~4.8% and latency ~3200ms, meeting SEV-1 criteria: critical user-facing flow broken with no workaround.

## Summary

Booking-service latency and error rate degraded significantly beginning 2026-06-01T14:25:00+00:00, 5 minutes after a deploy enabling payment eligibility enrichment. Payment eligibility lookups began timing out, causing booking requests to exhaust retries and fail; the booking queue backlogged from 12 messages to 545 messages, and appointments-db CPU rose from 38% to 92%.

- affected services: booking-service, payment-service, appointments-db, booking-queue
- customer impact: Users attempting to book appointments via POST /book experience request failures or timeouts exceeding 3+ seconds. Multiple booking attempts fail after exhausting internal retries, preventing users from completing appointment reservations.
- window: 2026-06-01T14:05:00+00:00 → ongoing (alert.triggered_at (2026-06-01T14:35:00+00:00) minus 30m lookback; deviated series not recovered -> ongoing)

## Ranked hypotheses

### 1. Payment eligibility enrichment feature flag caused exponential load on appointments-db, saturating connection pool and query latency — confidence: medium

Deploy 2026.06.01-1420 enabled the payment_enrichment feature flag at 2026-06-01T14:20:00+00:00, introducing synchronous eligibility enrichment queries on the booking-service POST /book critical path. These queries hit appointments-db eligibility_rules table, which degraded from 40ms baseline to 2150–2200ms (54–55x slowdown) within the incident window. Connection pool saturation (82% at 14:30:27, 97% at 14:34:44) and CPU elevation (71% at 14:30:00, peaking 93% at 14:50:00) reduced query throughput, causing payment-service check_eligibility to timeout at 2000ms threshold (observed at 14:31:08, 14:33:21, 14:55:46), which triggered booking-service retry loops (up to 5 attempts with no backoff per evidence_0d09fed9e4), exhausting retries and failing user bookings (evidence_3ab8c80bd9, evidence_ef577f6f8f). Queue backlog accumulated from 12 to 545 messages as failed eligibility lookups prevented booking completion, consistent with documented 'retry amplification' failure mode (evidence_e1bad1c80f).

### 2. Stripe API latency or availability issue exacerbated payment enrichment path bottleneck — confidence: medium

If Stripe API response times elevated or timeout behavior changed, payment-service check_eligibility would block longer, directly impacting booking-service latency. Payment-service p95 latency crossed 2x threshold at 14:25:00 (5 minutes after deploy, evidence_b6bb0d6a0b) and peaked at 2450ms (11.1x baseline) at 14:50:00. However, payment-service latency elevation temporally precedes appointments-db CPU saturation (14:25:00 vs. 14:30:00), consistent with a third-party dependency (Stripe) becoming the primary bottleneck. If Stripe latency is sustained (no spike in error rate reported), then payment-service would retry or queue calls, accumulating load on appointments-db connection pool via eligibility enrichment queries.

### 4. Retry amplification loop without exponential backoff caused unbounded queue growth and secondary saturation of appointment-db — confidence: low

Booking-service implements retry logic with a maximum of 5 attempts and no backoff (evidence_6faf946cf9, evidence_0d09fed9e4). When eligibility enrichment queries began timing out at 2000ms threshold at 14:31:08 (evidence_4d4a8dca3c), booking requests initiated retries. Each retry invoked the full eligibility enrichment path, re-querying appointments-db. Retries without backoff created a resonance effect: failed requests re-submitted immediately, each contributing to database load at the same moment, preventing query latency from recovering. Booking queue depth escalated from 35 messages at 14:25:00 (2x baseline) to 260 messages at 14:36:30 (21.7x baseline) to 545 messages at 15:00:00 (45.4x baseline), accumulating as eligibility lookup failures prevented booking completion. This queue backlog is documented in the runbook as exceeding safe operating range at >200 messages (evidence_df138b05ae). Sustained retry load compounded appointments-db saturation (97% connection pool at 14:34:44, CPU 93% at 14:50:00), preventing recovery and locking in multi-second latencies (2150–2200ms eligibility queries). The retry pattern persisted across the entire incident window (observed at 14:29:14, 14:31:08, 14:32:36, 14:35:52, 14:38:09, 14:45:17, 14:50:33 per evidence_6faf946cf9), indicating exponential backoff was absent.

- rubric: 3 aligned signal(s), timing aligned, 2 conflict(s)

### Appointment Booking Service Degradation: Impact of Payment Eligibility Enrichment Feature Enablement (June 1, 2026, 14:25–ongoing UTC)

Starting at 2026-06-01T14:25:00+00:00, the booking-service POST /book critical path experienced significant latency and error rate elevation. The degradation began approximately 5 minutes after deployment 2026.06.01-1420 rolled out at 2026-06-01T14:20:00+00:00, which enabled the payment_enrichment feature flag in booking-service. The feature flag introduced synchronous eligibility enrichment queries on the booking request path. These enrichment queries, targeting the appointments-db eligibility_rules table, experienced severe latency degradation from a baseline of 40ms to 2150–2200ms (a 54–55x slowdown), as evidenced in distributed traces and database slow-query logs (evidence_55c0d03512, evidence_66d868791e, evidence_525e4ad1e8). The appointments-db became the primary bottleneck: connection pool saturation climbed to 97/100 (97% utilization at 2026-06-01T14:34:44+00:00, evidence_90d23427ad), and CPU utilization elevated from 38% baseline to 92–93% peak (2.4x baseline, evidence_340a8a0871). When eligibility enrichment queries exceeded the 2000ms timeout threshold, booking-service initiated automatic retries. Booking-service retry logic is configured for up to 5 attempts with no exponential backoff (evidence_6faf946cf9, evidence_0d09fed9e4), causing retry traffic to re-submit immediately upon timeout, amplifying database load. Booking requests that exhausted all 5 retry attempts failed entirely, observed at 2026-06-01T14:35:52, 14:38:09, and 14:45:17 (evidence_3ab8c80bd9, evidence_ef577f6f8f).
